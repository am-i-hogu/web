<Title>Storybook 작성 양식 가이드</Title>
<Subtitle>공용 컴포넌트 스토리를 일관되게 작성하기 위한 기준 문서</Subtitle>

## 목적

- 신규 컴포넌트 추가 시 문서 품질을 일정하게 유지합니다.
- 누가 작성해도 같은 위치, 같은 구조, 같은 설명 톤을 유지합니다.
- 구현과 문서를 분리해 유지보수 비용을 줄입니다.

## 파일 위치 규칙

- 스토리 파일 위치: `storybook/stories/ui/<component>.stories.tsx`
- 컴포넌트 구현: `src/shared/ui/*` 또는 `src/features/*/ui/*`
- 공용 토큰/스타일: `src/shared/styles/*`

## 기본 작성 템플릿

```tsx
import type { Meta, StoryObj } from "@storybook/react";
import { Component } from "@/shared/ui";

const meta = {
  title: "UI/Component",
  component: Component,
  tags: ["autodocs"],
  args: {},
  argTypes: {},
  parameters: {
    layout: "centered",
  },
} satisfies Meta<typeof Component>;

export default meta;
type Story = StoryObj<typeof meta>;

export const Playground: Story = {};
```

## 작성 규칙

1. `title` 네이밍은 `UI/<컴포넌트명>` 형태를 기본으로 사용합니다.
2. 스토리는 최소 `Playground` 1개 + 핵심 상태 스토리를 추가합니다.
3. `args`는 가장 자주 쓰는 기본 상태를 기준으로 설정합니다.
4. 문구 설명은 짧고 기능 중심으로 작성합니다.
5. 인터랙션 예시는 실제 사용 흐름(토글, 선택, 닫기)을 재현합니다.

## 필수 확인 항목 (Checklist)

1. Controls에서 핵심 props가 조작 가능한가
2. `disabled`/`error`/`empty` 등 주요 상태가 분리되어 있는가
3. 모바일/앱뷰 기준 레이아웃이 깨지지 않는가
4. 아이콘/텍스트 컬러가 디자인 토큰을 따르는가
5. 문구와 스토리 이름이 컴포넌트 역할과 일치하는가

## 권장 스토리 구성

1. `Playground`: 기본 조합 확인
2. `State*`: 상태별 분리 (`Disabled`, `Selected`, `Error` 등)
3. `With*`: 조합 시나리오 (`WithIcon`, `WithDescription` 등)
4. `Flow*`: 단계형 UI의 흐름 시나리오 (예: 탈퇴 플로우)

## Do

- 디자인 토큰 클래스 사용 (`text-*`, `bg-*`, `border-*`)
- 가능한 한 컴포넌트 props만으로 상태를 표현
- 스토리 설명은 현재 스토리가 무엇을 검증하는지 명확히 작성

## Don't

- 스토리에서 임시 인라인 스타일로 UI를 강제하지 않기
- 실제 제품 로직(API, 전역 상태)을 스토리에 직접 연결하지 않기
- 비슷한 상태를 이름만 바꿔 중복 생성하지 않기

## 문서 업데이트 원칙

- 컴포넌트 API가 변경되면 같은 PR에서 스토리도 함께 수정합니다.
- 공용 컴포넌트는 구현 완료 후 Storybook 문서까지 작성되어야 완료로 판단합니다.
- 본 가이드는 팀 합의에 따라 업데이트하며, 변경 시 Overview에 요약을 남깁니다.
